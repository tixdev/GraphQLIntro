GraphQL

A Modern Approach to APIs

What is GraphQL?

A query language for APIs + a runtime to execute queries

📜

Origin

2012 Created at Facebook for mobile app needs
2015 Open-sourced to the community
2019 Moved to GraphQL Foundation (Linux Foundation)
🎯

Definition

A specification for how clients request data from a server. The client describes exactly what it needs, and gets back exactly that — nothing more, nothing less.

// Client asks for exactly what it needs
{ user(id: 1) { name, email } }

// Server returns exactly that
{ "user": { "name": "Alice", "email": "a@b.com" } }
🚫

What it is NOT

✗ Not a database
✗ Not a replacement for REST everywhere
✗ Not an ORM
✓ A contract between client and server

How it Works

Three pillars: Schema, Query, Resolvers

1

The N+1 Problem

The most common GraphQL performance pitfall

⚠️ PROBLEM Naive Implementation 
// This innocent query...
{
  users {
    name
    posts { title }
  }
}

// ...generates N+1 SQL queries:
SELECT * FROM users          // 1 query
SELECT * FROM posts WHERE user_id=1 // +N
SELECT * FROM posts WHERE user_id=2
SELECT * FROM posts WHERE user_id=3
// ... one per user!
⚠️ 100 users = 101 queries to the database
✅ SOLUTION DataLoader Pattern 
// HotChocolate DataLoader
public class PostsByUserLoader
  : BatchDataLoader<int, Post[]>
{
  protected override async Task
    LoadBatchAsync(
      IReadOnlyList<int> keys)
  {
    // ONE query for ALL users:
    SELECT * FROM posts
      WHERE user_id IN (1,2,3...)
  }
}
✅ 100 users = 2 queries (batched)
101 → 2
queries with DataLoader batching
